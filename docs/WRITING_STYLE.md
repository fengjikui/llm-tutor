# 写作风格约定

这套教程不是论文复述，也不是 API 文档。每一章都要让读者完成三件事：

1. 知道这个概念为什么出现。
2. 能读懂最小实现。
3. 能跑一个实验，并解释结果为什么大致合理。

## 讲解方式

- 先讲问题，再讲结构，最后讲公式。
- 每个公式都要配一句自然语言解释。
- 代码注释要解释 shape、训练阶段和容易误解的变量。
- 避免堆名词。第一次出现的术语必须给直觉解释。
- 不为了历史完整性讲太多过时模型，除非它能帮助理解 Transformer 或 LLM。

## 学习者视角

每章开头必须有“本章学习契约”，至少回答四件事：

- 这一章新增什么概念。
- 实验要验证什么。
- 实验不验证什么，避免读者过度解读。
- 跑完实验应该优先看哪些输出。

每个关键过渡点要主动替学习者发问，例如：

- 分类为什么转到回归讲梯度。
- CNN 为什么出现在 LLM 学习路线里。
- cross-attention 到 self-attention 到底改了什么。
- SFT 既然还是 next-token loss，为什么会改变回答风格。
- PPO/DPO/GRPO 的 toy bandit 和真实 LLM 训练有什么边界。

这些问题不要集中塞到最后的 FAQ，而是尽量放在读者刚要误解的位置。

## 实验要求

- 每个实验都必须能用一条命令运行。
- 默认参数要适合本地 CPU 快速 smoke test。
- 训练输出至少包含 loss 和一个可解释指标。
- 教程中要解释“这个实验能证明什么”和“它不能证明什么”。
- 如果实验使用教学抽象，例如 bandit 代替语言模型，要写清楚它对应真实系统的哪一部分，也要写清楚省略了什么。
- 训练型脚本应优先支持 `--output-dir`，保存 `config.json`、`metrics.jsonl`、`summary.json` 和 `stdout.log`；checkpoint 或模型权重只在确实有复盘价值时保存。

## 代码风格

- 优先清晰，不优先极致性能。
- 训练循环显式写出：forward -> loss -> backward -> step。
- 后期可以使用 Hugging Face / TRL，但核心数据格式和 loss 要先手写清楚。
